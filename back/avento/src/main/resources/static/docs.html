Avento - Documentação Técnica

Atualizado em 18/07/2026Local-firstAuth + Projetos + Voz + FerramentasComfyUI para mídia

O Avento é um assistente local para projetos de software e fluxos pessoais. A meta atual é ele trabalhar como um parceiro de desenvolvimento: analisar um projeto apontado pelo usuário, dar feedback em tempo real, usar ferramentas locais com segurança, ler e editar arquivos autorizados, rodar validações permitidas e conversar por voz.

O sistema roda com backend Spring Boot, frontend React/Vite, login local com JWT em cookie HttpOnly, PostgreSQL e Redis Stack via Docker/Colima, Ollama nativo no macOS, análise local de projetos, RAG, backups de arquivos, WebSocket para voz e serviços opcionais de Whisper.cpp/Piper.

Subir Tudo com Um Script

O comando principal do projeto agora é:

./scripts/dev-up.sh

Se você estiver dentro da pasta scripts/, use:

./dev-up.sh

Esse script checa dependências, reconcilia containers antigos sem perder os volumes persistentes, sobe PostgreSQL e Redis Stack via Docker/Colima, confirma Ollama, mata processos antigos do próprio Avento, sobe ComfyUI, backend e frontend e roda o smoke test autenticado. O terminal mostra as URLs, a credencial root local e os logs ao vivo; os mesmos logs também ficam em tmp/dev/. Cada rodada do agente registra início, conclusão ou falha, e o ComfyUI informa fila, progresso a cada 10 segundos e arquivo final. Para encerrar, use Ctrl+C; o script finaliza os processos que ele abriu.

Arquitetura Atual

Frontend

React + Vite em front. Conversa, seleção de workspace, árvore de arquivos, streaming do chat e controles de voz.

Backend

Spring Boot em back/avento/src/main/java/com/avento. Orquestra LLM, ferramentas, arquivos, projetos, RAG, voz e segurança local.

IA local

Ollama nativo no host. O agente usa /api/chat para streaming e chamadas de ferramenta.

Infra local

PostgreSQL e Redis Stack via Docker/Colima. PostgreSQL guarda os dados duráveis; Redis transporta jobs e eventos e acelera contexto, RAG e voz.

O frontend usa proxy do Vite para /api e /ws. Assim, o navegador conversa com o backend sem precisar expor a máquina inteira na rede.

Memoização: Sidebar e bolhas de mensagem

Implementado Sidebar e MessageBubble agora usam React.memo — antes, qualquer re-render do Home (inclusive o nível de áudio do microfone atualizando ~12x/s) re-renderizava a sidebar inteira e toda bolha de mensagem da conversa, mesmo sem relação nenhuma com o que mudou. Pré-requisito corrigido junto: handleChunkReceived mutava o objeto da última mensagem no lugar durante o streaming, o que teria quebrado a atualização visual do texto assim que a bolha virou memoizada — agora cria um objeto novo a cada chunk. Handlers passados como prop pra esses dois componentes (onOpenDiff, onOpenMedia, startNewChat, handleLoadChat) foram estabilizados com useCallback, já que uma prop de função nova a cada render invalida o memo mesmo com as outras props iguais.

Redis, Jobs e Streaming

O chat não mantém mais o trabalho preso ao mesmo POST do navegador. O frontend cria um job com POST /api/ai/runs, recebe o runId e acompanha GET /api/ai/runs/{runId}/events por SSE autenticado.

  1. Persistência: job e Outbox são salvos juntos no PostgreSQL.
  2. Publicação: o dispatcher envia apenas jobId e runId para avento:jobs:agent.
  3. Execução: AgentRunWorker consome com o group avento-agent-workers e chama Ollama, ferramentas MCP ou ComfyUI.
  4. Feedback: texto, thinking, ferramentas e estados entram em avento:events:{runId}, expiram após o TTL configurado e voltam ao chat por SSE sem reler eventos de outras execuções.
  5. Confirmação: o worker executa XACK depois de persistir o estado final.
PostgreSQL continua sendo a fonte oficial

Redis não é a única cópia da conversa. As últimas 20 mensagens ficam em avento:context:{userId}:{chatId} por até 24 horas; em cache miss ou reinício, o backend reconstrói a janela pelo PostgreSQL. Imagens, PDF, vídeo, áudio e base64 não entram nos Streams.

Jedis é necessário nesta versão

O Spring Data usa client-type: jedis porque o Spring AI 1.1.8 exige um JedisConnectionFactory para criar o VectorStore. Com Lettuce, o backend falha na criação do RagService.

Cancelamento e exclusão

POST /api/ai/runs/{runId}/cancel grava a solicitação e descarta a execução reativa ativa. Apagar o chat cancela runs, remove jobs, Outboxes e cache, além das mensagens e mídias já tratadas pelo fluxo de exclusão.

Watchdog contra execução travada

O limite total padrão continua em 15 minutos, mas o worker também observa atividade. Se nenhum token, evento ou progresso surgir por 3 minutos, o run vira FAILED, o SSE encerra e o próximo item da fila pode executar. Respostas baseadas no bloco [Project Analysis] também não são descartadas para uma segunda tentativa cega com todas as ferramentas MCP.

Configuração, comandos redis-cli, consultas SQL e diagnóstico completo: docs/REDIS_EXECUTION.md.

Agente, Projetos e Feedback em Tempo Real

O Avento agora tenta agir como um assistente de projeto, não apenas como chat. Quando um workspace é selecionado, o frontend injeta no prompt o contexto autorizado da máquina e a análise do projeto. Se o usuário colar um caminho absoluto como /Users/.../meu-projeto, o app tenta autorizar, carregar a árvore e analisar automaticamente.

  • Análise de projeto: detecta stack, scripts, entrypoints, arquivos importantes, riscos e recomendações.
  • Ferramentas MCP/locais: o agente pode executar ferramentas com limite de rodadas e eventos de timeline.
  • Comandos seguros: qualquer comando npm ou npx é permitido por inteiro (não é mais uma lista fixa de subcomandos — o usuário já sabe o que tá pedindo, não precisa ensinar cada ferramenta npm existente pro Avento). mvn {test, package, verify}, git {status, diff, log --oneline} e docker compose {ps, up -d, down, logs} continuam na lista fixa. Todo comando de terminal, de qualquer lista, ainda exige aprovação visual antes de rodar — ninguém executa nada sem você ver e confirmar. O ProcessBuilder recebe os argumentos já separados (sem invocar shell), então não há risco de injeção por ;, &&, | etc. mesmo com a lista mais aberta.
  • Arquivos: leitura/escrita somente em pastas autorizadas pelo usuário, com backup automático antes de alterar.
  • Histórico com contexto: conversas salvas persistem as pastas do projeto; ao abrir pelo histórico, o app revalida se cada pasta ainda existe antes de restaurar análise, árvore e painel lateral.
  • Roteamento casual: saudações e conversas curtas como oi, bom dia, tudo bem e como você está não recebem contexto de projeto, não disparam RAG e não executam ferramentas.
  • Filtro de tool markup: pseudo-chamadas como {function <directory_tree> ...} são tratadas internamente e não devem aparecer como resposta do chat.
  • Resposta local consciente: o system prompt foi ajustado para evitar a resposta genérica "não tenho acesso" quando há contexto local autorizado.
  • Streaming isolado por conversa: cada execução mantém seu próprio chatId, AbortController e rascunho. Trocar de chat deixa a origem trabalhando em segundo plano sem mover Thinking, resposta, eventos, mídia ou voz para a conversa aberta; ao voltar, o rascunho é restaurado. O botão de enviar vira parar apenas no chat que está respondendo e cancela somente os streams daquela conversa.
  • Fila de mensagens: mandar uma mensagem enquanto o Avento ainda está respondendo não descarta mais o texto — ela entra numa fila visível acima do campo de digitação, com prévia do texto e botão pra remover, e é enviada automaticamente assim que a resposta atual terminar. A fila é só da conversa aberta: trocar de chat ou começar uma nova descarta o que estava na fila (com aviso) em vez de deixar a mensagem vazar para a conversa seguinte.
  • Árvore de arquivos isolada por conversa: fileTree (a árvore mostrada na sidebar) só era sobrescrita quando o chat novo tinha projeto — trocar para um chat sem projeto, ou começar um novo, deixava a árvore/seleção de arquivos do chat anterior visível. clearFileTree() agora limpa isso no início de startNewChat e handleLoadChat, antes de qualquer carregamento condicional.
  • Indicador de "ainda pensando": os pontinhos de digitação agora aparecem em duas situações que antes ficavam mudas — quando a única coisa gerada até agora é um bloco plan (que não aparece no chat) e durante o intervalo entre chamadas de ferramenta, quando já existe texto visível mas o agente ainda está trabalhando.
  • Contagem de tokens real: o contador de tokens exibido no rodapé de cada resposta contava chunks de streaming recebidos, sem relação real com tokens (um chunk pode ter zero, um ou vários). Agora usa o eval_count real que o Ollama reporta ao final de cada rodada (evento agent.tokens.usage); a estimativa por chunk continua só como número que se move durante o streaming, substituída pelo valor real assim que cada rodada termina.
[Workspace Roots] + [Project Analysis] + conversa do usuário -> AgentService -> Ollama + ferramentas locais

Diretórios grandes ou de runtime, como whisper.cpp, piper_tts, tmp e logs, são ignorados na análise para evitar ruído.

Arquitetura Planejada: Hub + Agente de Dispositivo

Planejado — ainda não implementado

Hoje o backend roda tudo no mesmo processo: o LLM, o chat, e também a execução de ações físicas (abrir app, tirar print, automação macOS via SystemAutomationService). Isso só funciona porque backend e usuário estão na mesma máquina. O próximo passo de arquitetura separa essas duas responsabilidades:

Hub (este backend)

Continua central, rodando num PC: Ollama/LLM, orquestração do agente, RAG, chat, timeline, aprovações, políticas, e ferramentas ligadas a código/projeto (git, terminal, banco de dados).

Agente Windows

Processo leve pareado ao Hub. Executa: abrir/fechar apps, notificações, captura de tela, seletor de pasta nativo.

Agente Linux

Mesma responsabilidade do Agente Windows, com implementação própria (xdg-open, notify-send, zenity/kdialog).

Agente macOS

Sucessor do SystemAutomationService atual: AppleScript/osascript, Finder, apps, print de tela — extraído do Hub para rodar como agente pareado.

Agente Android

App nativo com Accessibility Service, conectado ao Hub por WebSocket. Vira um alvo de automação real, não apenas um cliente de leitura.

Fluxo de uma ação física: o agente de IA decide chamar uma ferramenta como capture_screen; em vez de executar no processo do Hub, o pedido é despachado para o Agente do dispositivo ativo daquele usuário, que executa localmente e devolve o resultado. A aprovação continua acontecendo na conversa central, nunca silenciosamente no dispositivo remoto.

Hub (LLM + aprovação) --[WebSocket pareado]--> Agente do dispositivo --[executa]--> devolve resultado --> Hub continua o raciocínio

Cada dispositivo precisa de pareamento explícito (código/QR) e recebe um token próprio, revogável a qualquer momento. Um usuário pode ter vários dispositivos pareados; a conversa escolhe qual é o "dispositivo alvo" de cada ação.

Ponto de atenção

O canal Hub↔dispositivo é uma superfície de rede nova, separada da guarda loopback-only que protege a API HTTP atual. Precisa nascer com token por dispositivo e transporte criptografado (WSS) desde o primeiro protótipo — não é algo para adicionar depois.

Toda ferramenta desconhecida agora exige aprovação por padrão

Correção de segurança Antes, ToolCapabilityRegistry.requiresApproval() liberava sem aprovação qualquer ferramenta que não estivesse cadastrada nas ~20 nativas — incluindo tudo que vem de MCP externo (Desktop Commander, Automator, Playwright, Puppeteer, e as 5 novas acima). Isso foi invertido: ferramenta desconhecida agora exige aprovação por padrão, e só roda livre se estiver explicitamente marcada AUTO no registro.

Orquestrador único e SDK MCP oficial

Implementado Toda solicitação cria um runId no AgentOrchestrator, atravessa o mesmo gateway de ferramentas e fica em RUNNING, AWAITING_APPROVAL, COMPLETED, FAILED ou CANCELLED. Servidores externos usam o SDK Java oficial do MCP por stdio; descoberta, colisão de nomes e execução deixam de depender do parser manual legado. O resultado de comandos, escrita de arquivo e scaffolds é verificado antes de o agente anunciar sucesso.

Use AVENTO_MCP_SDK_ENABLED=false apenas como fallback temporário e AVENTO_MCP_SDK_REQUEST_TIMEOUT=10s para ajustar o tempo de inicialização e chamada. Conexões repetidas são idempotentes e não reiniciam um servidor já ativo. O Desktop Commander continua no catálogo para conexão manual, sem bloquear a subida automática.

Geração de imagem: instrução direta, não só política

Corrigido generate_image nunca era mencionado em agent/instructions/tools.md — a única instrução vivia misturada dentro do texto de política de conteúdo (agent/policies/*.md), que fala de critérios como "nudez artística vs sexualizada". O modelo tendia a "avaliar" esses critérios em texto em vez de simplesmente chamar a ferramenta. Agora tools.md tem uma linha direta ("pedido de imagem → generate_image, decida internamente, nunca narre a política").

Política pública e configuração pessoal separadas

Atualizado agent/policies/maximum.md mantém uma política pública adequada para o repositório. Tradução, transcrição, resumo, revisão, classificação e análise de material fornecido continuam tratadas como transformação: preservam significado, intensidade, palavrões e linguagem adulta ou explícita, sem censura ou eufemismo.

Local Uma máquina pode substituir a política embutida com ~/.avento/policies/{modo}.md ou configurar outro diretório por AVENTO_AGENT_POLICY_OVERRIDE_DIR. O conteúdo pessoal não entra no Git; sem o arquivo local, o backend usa a política pública versionada.

Modelo As políticas e os procedimentos internos de imagem e tradução são escritos em inglês para melhorar a aderência de modelos locais. A interface e as respostas continuam acompanhando o idioma do usuário.

Exposição de ferramentas: baratas sempre visíveis, caras filtradas

Implementado generate_image e capture_screen (schema pequeno, alto valor) agora ficam sempre expostas ao modelo — não dependem mais de bater com uma palavra-chave. O filtro de intenção continua só para os clusters caros (MCP externo com dezenas de ferramentas cada). Complementa isso uma rede de segurança: se a primeira rodada não usar nenhuma ferramenta numa mensagem que não é conversa casual, o backend tenta de novo automaticamente com o conjunto completo de ferramentas, uma única vez por resposta.

Aviso de ação não confirmada O modelo pode descrever uma ação como concluída em texto ("✅ criado com sucesso") sem ter chamado nenhuma ferramenta de verdade — já aconteceu com pedidos grandes (ex.: gerar vários serviços NestJS de uma vez). Não dá pra impedir isso só por instrução de prompt, e tentar detectar a mentira no texto seria frágil. Em vez disso, o backend usa a única fonte confiável que existe — state.executedToolCalls, que só sobe quando uma ferramenta é de fato executada — e anexa um aviso visível na resposta sempre que o pedido era acionável (não é conversa casual) e o turno inteiro, já com o retry tentado, executou zero ferramentas.

Classificação de intenção por significado, não só palavra-chave

Implementado IntentEmbeddingClassifier usa o mesmo modelo de embedding do RAG (nomic-embed-text) para comparar o significado da mensagem do usuário com frases-exemplo por categoria (agent/heuristics/intent-examples.txt), em vez de exigir bater a palavra exata. Funciona como sinal adicional ao IntentRouter por palavra-chave — os dois sinais se somam (união, não substituição), e se o Ollama estiver indisponível ou lento, cai de volta sozinho para o classificador por palavra-chave sem quebrar nada.

Corrigido Achado de performance: IntentRouter.classify() era chamado uma vez por ferramenta dentro do loop de seleção de tools — com MCP externo conectado (dezenas de ferramentas), isso virava dezenas de chamadas de embedding redundantes ao Ollama pra mesma mensagem, a cada turno. Agora AgentService.selectToolsForCurrentRequest classifica a mensagem uma única vez e reusa o resultado (IntentProfile) pra todas as ferramentas do loop.

Aprovação por plano, não ação por ação

Implementado Antes de agir em pedidos com mais de um passo, o agente escreve um plano (ver agent/instructions/execution.md). Ao aprovar a primeira ação, o backend libera as próximas ações da mesma resposta automaticamente — sem novo popup — exceto delete_file, terminal_stop e close_app, que sempre pedem confirmação própria mesmo com o plano já aprovado. A liberação vale só para aquela resposta (runId); a próxima mensagem do usuário começa sem nada pré-aprovado.

Atualizado O plano não aparece mais como texto solto no chat — o modelo escreve os passos dentro de um bloco ```plan```, o frontend remove esse bloco do balão de mensagem (mesmo padrão já usado pro bloco ```file-edit```) e mostra os passos numerados na aba "Tarefas e Contexto". A aba abre uma vez quando o plano aparece e respeita o fechamento manual durante o restante da execução, mesmo que novos chunks ou eventos de ferramenta atualizem o plano. O plano some da aba assim que uma mensagem nova é enviada.

Fallback O modelo local nem sempre usa o bloco ```plan``` pedido — às vezes escreve os passos como lista numerada solta mesmo. extractLooseNumberedList detecta esse padrão também e duplica na aba, mas não esconde do chat nesse caso (uma lista numerada solta pode ser só uma explicação normal, não necessariamente um plano de execução — só o bloco ```plan``` explícito é removido, por ser um sinal inequívoco).

Skills — procedimentos reutilizáveis via /nome

Implementado Um pedido comum e repetido não precisa fazer o modelo reconstruir o procedimento do zero. Arquivos .md em agent/skills/ são carregados pelo SkillRegistry; digitar /nome argumento injeta o procedimento no turno e mantém o mesmo modelo, Permission Engine e fluxo de ferramentas.

Pacote padrão O Avento inclui procedimentos para analisar, corrigir, diagnosticar, executar e criar projetos; manter Java/Spring, Spring Security, banco/Flyway, integrações MCP, jobs Redis/SSE, frontend, mídia, voz, RAG e dependências; ler documentos; traduzir fielmente conteúdo fornecido; consultar bancos; gerenciar memória, Git e Docker; pesquisar na web; finalizar mudanças e validar releases. Use /skills para ver os nomes atuais. Os procedimentos usam somente ferramentas que o Avento realmente oferece e exigem resultado confirmado antes de anunciar sucesso.

Skills compartilhadas As skills embutidas orientam o próprio Avento. Para Codex e outros agentes que trabalham neste repositório, AGENTS.md aponta para as versões detalhadas e validáveis em .agents/skills/.

Ativação automática Cada arquivo pode declarar uma linha Gatilhos:. As frases são normalizadas sem acentos e, quando mais de uma combinar, vence a mais específica. Mensagens casuais nunca ativam skill. A skill translate-content reconhece pedidos diretos de tradução e preserva o texto original no turno. Toda ativação emite skill.activated no painel de atividade.

Ferramenta declarada Além de Gatilhos:, o cabeçalho pode declarar Ferramenta: — a ferramenta que a skill comanda. A skill sempre passa pelo modelo (ele raciocina e decide a chamada), mas a ferramenta declarada é exposta com prioridade na seleção, imune às heurísticas de palavra-chave que antes desviavam pedidos de vídeo para generate_image. Assim /generate-video chega ao generate_video e /generate-image ao generate_image, mesmo quando o texto do pedido menciona a outra mídia. generate-video ganhou gatilhos de linguagem natural; generate-image permanece sem gatilhos porque "gera uma imagem" já é capturado pelo detector direto de imagem.

Descoberta /skills lista todas as skills instaladas com descrição, sem chamar o modelo. Nome de skill que não existe responde na hora com essa mesma lista. Digitar / no campo de mensagem abre um autocomplete com as skills disponíveis (↑↓ navega, Enter/Tab escolhe, Esc fecha).

Criação pela interface O botão de raio (⚡) na barra superior abre o gerenciador de skills: lista as instaladas (com selo "sistema" ou "sua"), permite criar novas (nome, descrição, gatilhos de ativação automática e o procedimento) e remover as suas. Skills criadas pela UI ficam em data/skills/*.md (fora do git) e sobrevivem a reinício do backend — endpoints em /api/skills (GET/POST/DELETE), skills embutidas do sistema não podem ser alteradas nem removidas.

Heurísticas de mensagem em arquivo, não em código

As listas de palavras usadas para detectar conversa casual e para decidir quais ferramentas expor por mensagem agora vivem em back/avento/src/main/resources/agent/heuristics/*.txt (casual-phrases.txt, project-action-words.txt, intent-keywords.txt), carregadas por HeuristicWordLists. Editar uma palavra-chave não exige recompilar nem mexer no AgentService.java ou IntentRouter.java.

Segurança Local

O Avento está configurado como ferramenta local-first. O backend fica preso em 127.0.0.1 por padrão, bloqueia uso não-loopback quando a configuração segura está ativa e exige login local quando avento.auth.enabled=true.

  • Loopback: servidor e CORS aceitam somente origens locais por padrão.
  • Login root local: o profile local cria/atualiza o e-mail definido por AVENTO_AUTH_ROOT_EMAIL (padrao admin@avento.local) com role ROOT. A senha deve ser definida por AVENTO_AUTH_ROOT_PASSWORD; esse seed pode ser desligado por AVENTO_AUTH_ROOT_ENABLED=false.
  • Cookie de dev: access JWT vai em cookie HttpOnly com SameSite=Lax; em dev Secure=false porque o ambiente roda em HTTP local.
  • Refresh interno: o navegador não recebe refresh token. O backend usa sid/jti do access token para achar a sessão, rotacionar o jti e registrar auditoria.
  • Axios no frontend: chamadas REST /api usam um client Axios compartilhado com withCredentials=true; streams de chat usam fetch apenas para ler ReadableStream.
  • Refresh coordenado: o frontend compartilha uma única chamada de /api/auth/refresh entre requests simultâneos para evitar corrida na rotação do jti.
  • Cookie como credencial única: a sessão é validada pelo access JWT em cookie HttpOnly; o frontend não lê nem envia token de autenticação em JavaScript.
  • Histórico por usuário: conversas e mensagens de /api/chats são filtradas pelo usuário autenticado; um perfil não carrega nem altera chats de outro.
  • Feedback de autorização: respostas 401 e 403 retornam JSON com status, mensagem e rota; o frontend exibe sessão vencida com revalidação ou snackbar de acesso negado.
  • Autorização de workspace: o usuário precisa liberar uma pasta antes de ler, listar ou escrever arquivos.
  • Aprovação por voz: ações pendentes podem ser aprovadas ou canceladas falando frases como “aprovo por 24 horas nesse projeto” ou “cancela”; permissões temporárias ficam vinculadas ao projeto, ferramenta e recurso/comando.
  • Backup antes de escrita/remoção: cada alteração ou remoção controlada cria cópia restaurável.
  • Restore: endpoint dedicado para restaurar versões salvas.
  • Comandos restritos: nada de shell livre para o modelo; apenas validações aprovadas.
  • Erros controlados: falhas do agente viram eventos de stream em vez de quebrar a UI com 500 genérico.

Para expor o Avento em rede no futuro, será necessário habilitar HTTPS, usar segredo JWT forte, revisar CORS, limitar permissões por workspace e acompanhar auditoria de ações.

Voz em Tempo Quase Real

Indicador visual de captura Um medidor de nível de áudio (barras que reagem em tempo real) aparece ao lado do botão de microfone sempre que a gravação por clique ou o modo de voz em tempo real estiverem ativos — confirmação visual real de que o som está chegando, não só um ícone estático de "gravando". Implementado com AnalyserNode da Web Audio API, atualizado a cada ~80ms.

Existem dois fluxos de voz no frontend:

Controle Como funciona Quando usar
Microfone tradicional Grava tudo via MediaRecorder e só transcreve quando o usuário para a gravação. Bom para ditar uma mensagem maior e revisar antes de enviar.
Conversa em tempo quase real Abre /ws/voice, detecta silêncio, envia a frase WebM completa, manda flush, recebe transcrição e envia ao chat automaticamente. Bom para falar direto com o Avento sem ficar clicando parar a cada frase.

No backend, o WebSocket recebe o WebM completo de cada frase com buffer ampliado para áudio, converte com FFmpeg e transcreve localmente com Whisper.cpp. O Avento detecta idioma localmente, aceita apenas idiomas configurados como permitidos e refaz em português quando o Whisper tenta sair para outro idioma ou alfabeto. Ele também passa um prompt local de vocabulário e --carry-initial-prompt para preservar nomes como VS Code, Brave, Finder e termos de projeto em comandos curtos. O frontend recebe JSON como {"type":"transcript.final","text":"bom dia","language":"pt"}.

O modo conversa também tem interrupção por fala, ou barge-in: se o usuário começa a falar enquanto o Avento está respondendo por voz, o frontend corta o áudio atual imediatamente, aborta a geração em andamento quando necessário e mantém a captura ativa até detectar cerca de 2 segundos de silêncio, para enviar a nova fala com o contexto recente da conversa.

O TTS usa cache opcional no Redis para reaproveitar WAVs de frases iguais e reduzir latência. Se o Redis estiver indisponível, o Piper continua sendo usado normalmente. O frontend envia o idioma detectado da resposta para /api/voice/tts, e o backend escolhe piper-model-pt, piper-model-en ou piper-model-es quando configurados. Para inglês soar natural, use um modelo Piper em inglês em avento.voice.piper-model-en; sem isso, o fallback PT-BR continua funcionando, mas pode soar como leitura com sotaque. Caminhos relativos de Piper e Whisper são resolvidos pela raiz indicada em AVENTO_PROJECT_ROOT, que o dev-up.sh configura automaticamente. Se o projeto tiver sido movido, o backend também contorna shebangs antigos do virtualenv ao iniciar o Piper pelo Python local correspondente.

Limite atual

Esse fluxo é frase/silêncio-based. Ele não é streaming neural palavra por palavra. Para chegar nesse nível, o próximo passo é VAD mais preciso, pipeline WebRTC ou Whisper streaming dedicado.

Ollama, Colima e Aceleração no Mac

No macOS, a recomendação é rodar o Ollama nativo no host, fora do Docker/Colima. Assim ele consegue usar aceleração Metal em Apple Silicon. Rodar Ollama dentro de container no Mac geralmente não entrega passthrough de GPU/Metal e tende a cair para CPU, ficando bem mais lento.

Para o agente com ferramentas locais habilitadas, prefira qwen3:8b. Nos testes locais ele respondeu saudações em português sem acionar ferramentas e chamou ferramenta apenas em pedido explícito. O llama3.2 continua útil como fallback leve, mas pode confundir saudações curtas com chamadas de ferramenta quando muitas tools estão expostas. A interface marca modelos pesados no seletor.

O backend compacta histórico antigo, remove prompts de sistema duplicados do frontend e envia apenas as ferramentas relevantes para o pedido antes de chamar o modelo. Isso reduz warnings como truncating input prompt limit=4096 prompt=17136 quando a conversa acumula contexto de projeto, ferramentas e validações. A propriedade avento.agent.num-ctx=8192 fica disponível para provedores/endpoints que respeitam essa opção.

Use Docker/Colima para PostgreSQL e Redis Stack, mantendo Ollama no host:

colima start docker compose up -d ollama serve

Modelos recomendados:

ollama pull qwen3:8b ollama pull qwen2.5vl:7b ollama pull llama3.2 ollama pull nomic-embed-text

Se algum serviço dentro do Colima precisar chamar o Ollama do host, use host.docker.internal ou host.lima.internal, dependendo da configuração do Colima.

Leitura e Geração de Imagens e Vídeos

As responsabilidades são separadas: modelos multimodais do Ollama leem e interpretam anexos PNG, JPEG e WebP, enquanto o ComfyUI executa os workflows locais de geração de mídia. A geração visual não exige workspace nem servidor MCP. Pedidos explícitos e descrições visuais autônomas com sinais suficientes de estilo e composição seguem diretamente para generate_image; pedidos para analisar, explicar ou melhorar um prompt continuam na conversa. Ao receber uma imagem, o Avento usa o modelo configurado em AVENTO_AGENT_VISION_MODEL e preserva o contexto visual na conversa.

A ferramenta generate_image envia o prompt e os parâmetros do header para o workflow compatível. O RealVisXL V5 é o padrão recomendado e usa SDXL em resolução nativa, VAE próprio, segundo passe e detailers. Uma imagem anexada pode orientar Composição por Depth/Canny ControlNet, Identidade por IP-Adapter ou Transformação por img2img; a pose possui um arquivo e uma força independentes. O Realistic Vision V6 permanece como alternativa SD 1.5 e o FLUX.2 Klein como opção destilada de uso geral. O controle Validar resultado compara localmente a saída ao pedido com o modelo visual e permite até duas tentativas corretivas; essa revisão aumenta o tempo e nunca bloqueia a imagem existente quando não puder ser executada. scripts/setup-comfyui-sdxl.sh instala RealVisXL, VAE, ControlNets, IP-Adapter e preprocessadores com downloads retomáveis e SHA-256, e o dev-up.sh executa esse preparo automaticamente salvo quando AVENTO_COMFYUI_SDXL_AUTO_INSTALL=0.

Tradução de prompt O encoder CLIP dos checkpoints SDXL entende apenas inglês e trunca o prompt em 77 tokens; um pedido em português chegava como ruído. Antes de montar o prompt, o ImagePromptTranslator traduz o pedido para inglês com o modelo de conversa (Ollama, think:false, tradução literal que não inventa detalhes), e qualquer falha mantém o pedido original. O planner também passou a colocar o conteúdo do usuário no início do prompt positivo, de modo que a truncagem do CLIP corte o reforço do planner, nunca o pedido. Controle por AVENTO_IMAGE_TRANSLATION_ENABLED, AVENTO_IMAGE_TRANSLATION_MODEL e AVENTO_IMAGE_TRANSLATION_TIMEOUT_SECONDS.

Presets por modelo Cada família de checkpoint tem seu ponto ideal de sampler, passos, CFG e resolução nativa. O ImageModelPresetCatalog resolve o preset do modelo selecionado a partir de comfyui/model-presets.json: sdxl-photoreal (RealVisXL/JuggernautXL/SDXL), flux2-klein destilado (4 passos, CFG 1.0), flux2-klein-base (16–24 passos, CFG 3.5–4.5) e stable-diffusion-1.5 como padrão. O seletor draft/balanced/quality escolhe a faixa dentro do preset e ajustes manuais sempre vencem — um CFG definido na mão sobrepõe o do preset. É possível sobrepor ou adicionar presets em ~/.avento/image-presets.json (AVENTO_IMAGE_PRESETS_FILE), relido a cada geração e sem entrar no Git.

FLUX.2 fiel ao pedido O encoder do FLUX.2 Klein é um LLM (qwen), que segue frases naturais e trata a sopa de tags do planner SDXL como ruído — era a causa das imagens ignorarem o pedido. O preset declara promptStyle: tags (SDXL/SD 1.5) recebe o prompt reforçado do planner; natural (FLUX.2) recebe o pedido traduzido como frase limpa. O destilado roda em 4 passos com CFG 1.0 por recomendação oficial; a variante base tem preset próprio para aderência máxima ao custo de mais tempo.

A ferramenta generate_video usa WAN 2.2 TI2V 5B. Ela pode animar a imagem mais recente do chat ou trabalhar apenas com texto e é executada de forma assíncrona, com progresso, estimativa e cancelamento no frontend. O workflow fica em back/avento/src/main/resources/comfyui/workflows/text-to-video-api.json; scripts/setup-comfyui-video.sh instala os modelos necessários com retomada automática.

O script scripts/dev-up.sh inicia e acompanha Ollama e ComfyUI junto com PostgreSQL, Redis, backend e frontend. O MarkItDown é validado durante a subida e conectado automaticamente pelo catálogo MCP. As mídias concluídas aparecem no chat, ficam vinculadas à conversa e são exibidas na seção de mídia do frontend. Imagens e vídeos podem ser minimizados em um cabeçalho compacto e expandidos novamente pela seta, no mesmo padrão do bloco de raciocínio. A seção Mídias da barra lateral também é recolhível e mostra apenas uma lista compacta do chat selecionado.

Saída visual O chat renderiza tabelas Markdown (GFM via remark-gfm), e o modelo pode responder com um bloco ui-preview autocontido para relatórios, dashboards e gráficos em SVG inline — sem rede, no mesmo iframe isolado dos protótipos. A skill visual-report orienta esse formato.

Exportar PDF A ferramenta generate_pdf converte Markdown (com tabelas) ou HTML em um PDF via openhtmltopdf, salvo em ~/Pictures/Avento Generated Images e vinculado à conversa (marcador [[avento-doc:...]], card de download no balão). É servido pelo mesmo endpoint de mídia e apagado junto com o chat.

Pesquisa na internet A skill /research declara Ferramentas: (plural — fetch, web_url_read, browser_navigate, browser_snapshot) e MaxRodadas:, que eleva o teto de rodadas só para a pesquisa (o global de 6 continua protegendo as demais conversas). O procedimento busca, extrai e sintetiza numa tabela ou relatório com fontes citadas.

Mockups e Protótipos de Interface

Pedidos de mockup de tela, mockup de interface, ui mockup, website mockup, wireframe ou protótipo ativam a skill prototype-interface antes do roteamento de geração de imagens. Esse fluxo não usa ComfyUI: o modelo produz um documento HTML autocontido em um bloco ui-preview e o navegador o renderiza localmente.

O resultado aparece recolhido como uma miniatura para preservar espaço no chat. Clique nela, ou use Enter/Espaço com foco, para abrir a prévia no próprio balão. Os controles alternam desktop, tablet e celular; o botão de expansão abre uma área maior. Scripts começam desativados e só recebem allow-scripts quando a interação é habilitada manualmente.

O HTML permanece no conteúdo persistido da mensagem, então o mockup volta com o histórico sem criar mídia adicional. Depois da aprovação explícita, o agente pode transformar a proposta nos componentes reais do workspace e validar a implementação com Playwright.

Documentos no Chat

O botão de clipe do composer aceita até quatro arquivos por mensagem: PDF, Word, Excel, PowerPoint, EPUB, ZIP, texto e formatos comuns de código. Cada arquivo pode ter até 50 MB.

O frontend envia o arquivo ao endpoint autenticado POST /api/documents/extract. O backend cria uma cópia temporária, lê texto simples diretamente ou converte formatos binários com o MarkItDown local e apaga a cópia ao terminar. A instalação padrão em .avento-tools/mcp é resolvida a partir da raiz do Avento, mesmo quando o Back é iniciado por outro diretório. O arquivo original não é movido nem alterado.

O nome e o contexto extraído ficam persistidos com a mensagem, portanto o Avento continua reconhecendo o documento ao reabrir o chat. Para preservar a janela do modelo local, o limite padrão é de 5.000 caracteres por documento e pode ser alterado por AVENTO_DOCUMENT_ATTACHMENT_MAX_CONTEXT_CHARS.

Endpoints Principais

Contrato das respostas JSON

As rotas REST JSON retornam um envelope com status, code e data. O status repete o HTTP real, o código é estável para clientes e os dados produzidos ficam em data. Uma consulta sem itens retorna data: [].

{
  "status": 200,
  "code": "SUCCESS",
  "data": []
}

Quando há erro, o próprio erro é o dado retornado. data inclui mensagem, rota, instante, traceId e erros de campo. O mesmo identificador aparece no header X-Trace-Id. O ControllerAdvice centraliza validação e exceções de domínio, enquanto o Spring Security reutiliza o mesmo formato em 401 e 403.

{
  "status": 400,
  "code": "INVALID_REQUEST",
  "data": {
    "message": "path is required",
    "path": "/api/projects/analyze",
    "timestamp": "2026-07-17T12:00:00Z",
    "traceId": "...",
    "errors": []
  }
}

O Axios compartilhado desembrulha o envelope para os componentes React. SSE, WebSocket, áudio e downloads usam corpo bruto e ficam fora desse contrato.

Validação e Diagnóstico

Comandos úteis para verificar o projeto:

./scripts/check-local-deps.sh ./scripts/smoke-local.sh mvn -f back/avento/pom.xml test cd front && npm run validate

O smoke test valida backend, frontend, autorização de workspace, análise de projeto, endpoint do Ollama e, opcionalmente, execução de comando seguro.

O aviso do Vite sobre bundle maior que 500KB ainda pode aparecer. Ele não bloqueia o app, mas pode virar tarefa futura de code splitting.

Próximos Passos Recomendados

Ordem sugerida para levar o Avento de ferramenta local single-machine para o modelo Hub + Agente de Dispositivo (ver seção de arquitetura):

Fase 0 — Protocolo Hub + Agente

Pareamento, autenticação por token, canal WebSocket criptografado, anúncio de capacidades por dispositivo. Tudo mais depende disso.

Fase 1 — Primeiro Agente (macOS)

Extrair o SystemAutomationService atual do Hub para um Agente macOS separado, provando o modelo antes de replicar para outros sistemas.

Fase 2 — Paridade de agente

Edição por diff e lista de tarefas visível — maior ganho de sensação de "trabalhar com o agente" pelo menor esforço.

Fase 3 — Ferramentas Hub-side

Git MCP primeiro, depois Fetch/Postgres/Docker — todas cross-platform por natureza, aprovação obrigatória mantida.

Fase 4 — Agentes Windows, Linux e Android

Réplica do padrão provado na Fase 1; Android ganha um agente nativo de verdade via Accessibility Service.

Fase 5 — Empacotamento, CI e alcance

Instaladores por sistema operacional, pipeline com matriz de build, i18n, modo de planejamento e comandos reutilizáveis.

  4. Persistir análises de projeto e comparar mudanças entre execuções.
